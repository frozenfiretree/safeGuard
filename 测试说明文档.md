# SafeGuard 测试说明文档

## 1. 文档目的

本文档用于指导测试组对 SafeGuard 内网敏感文件检测与溯源系统进行系统测试、回归测试和验收测试。

测试目标：

- 验证 Server、OCR Service、WebUI、Windows Agent、gRPC 上传服务是否能按预期协同工作。
- 验证 Agent 注册、心跳、配置拉取、文件扫描、文件上传、事件上报、敏感检测、敏感档案、版本溯源和归档下载等核心链路。
- 验证管理端 WebUI 的登录、页面展示、筛选、操作、下载和配置能力。
- 验证异常场景、安全鉴权、兼容性、性能和部署可用性。
- 为每轮测试输出可追溯的问题记录和验收结论。

## 2. 系统范围

### 2.1 被测组件

| 组件 | 路径或入口 | 测试重点 |
| --- | --- | --- |
| FastAPI Server | `server/main.py` | API、认证、数据写入、任务调度、静态 WebUI 托管 |
| 管理端 WebUI | `server/webui/index.html` | 登录、页面导航、数据展示、筛选、详情、操作反馈 |
| OCR Service | `server/ocr_service.py` | 健康检查、模型加载、图片/PDF/Office 内嵌图片识别 |
| gRPC Upload Server | `server/grpc_upload_server.py` | Agent 文件分块上传、断点状态、服务健康 |
| Windows Agent | `agent_core/`、`agent.py`、`dist/agent-service/SafeGuardAgent.exe` | 注册、配置、扫描、监控、上传、事件、升级 |
| 检测流水线 | `server/detection/` | 文件解析、规则命中、OCR 命中、LLM 复核门控 |
| 敏感档案 | `server/tracked_files.py` | 建档、版本、改名、修改、删除、下载 |
| 资产发现 | `server/asset_discovery.py` | 网段推断、ARP、ping/nmap 探测、端口扫描 |
| 部署脚本 | `deploy/`、`run_server.bat`、`run_ocr_service.bat` | Windows 开发启动、Linux 测试部署、Agent 服务安装 |

### 2.2 不在本轮默认范围内

除非测试计划特别说明，以下内容不作为常规验收阻塞项：

- 大规模生产级 PostgreSQL、Redis、MinIO、Celery worker/beat 集群化能力。
- OCR GPU 性能调优。
- 真实外部 LLM 供应商稳定性。
- 第三方杀毒、EDR、安全网关对 Agent 的拦截兼容。

## 3. 测试环境

### 3.1 推荐环境拓扑

| 角色 | 建议配置 | 说明 |
| --- | --- | --- |
| Server 测试机 | Linux 或 Windows 开发机，Python 环境完整 | 运行 FastAPI Server、gRPC Upload Server 和 WebUI |
| OCR 测试机 | 可与 Server 同机 | 默认监听 `127.0.0.1:8010` |
| Windows Agent 测试机 | Windows 10/11 或 Windows Server | 安装或前台运行 Agent |
| 浏览器 | Chrome / Edge 最新稳定版 | 测试 WebUI |
| 网络 | Server 与 Agent 互通 TCP 80/8000、50051 | `50051` 用于 gRPC 上传 |

### 3.2 默认端口

| 端口 | 服务 | 期望状态 |
| --- | --- | --- |
| `8000` | FastAPI Server | 可从浏览器或 curl 访问 |
| `8010` | OCR Service | 仅本机访问即可 |
| `50051` | gRPC Upload Server | Agent 可访问 |
| `80` | nginx 反向代理，Linux 部署时使用 | Agent 和浏览器可访问 |

### 3.3 默认测试账号和 Token

开发环境默认管理员 Token：

```text
dev-admin-token
```

生产或准生产测试环境必须使用部署方提供的强随机 Token，不允许继续使用默认 Token。

WebUI 支持以下输入形式：

```text
dev-admin-token
Bearer dev-admin-token
Authorization: Bearer dev-admin-token
X-Admin-Token: dev-admin-token
admin:password
```

### 3.4 测试数据目录建议

Windows Agent 测试机建议准备：

```text
C:\SafeGuardTest\
├─ normal\
├─ sensitive\
├─ office\
├─ image\
├─ large\
└─ rename-delete\
```

测试过程中不要选择系统目录、用户真实工作目录或包含真实隐私数据的目录作为扫描目录。

## 4. 测试准备

### 4.1 代码和工作区检查

在项目根目录执行：

```powershell
git status --short
python --version
python -m pip install -r requirements.txt
```

注意：

ocr模型文件（models.zip）要解压并放入server/artifacts/models目录下，否则ocr服务不能正常运行。

.doc、.ppt类型文档解析与高亮需先下载libreoffice。

**Windows 安装**

1. 从官方页面下载安装包：[LibreOffice Windows Download](https://www.libreoffice.org/download/Windows)。
2. 正常安装，默认路径通常是：

```
C:\Program Files\LibreOffice\program 
```

1. 把它加入系统 PATH。用管理员 PowerShell 执行：

```
$lo = "C:\Program Files\LibreOffice\program" [Environment]::SetEnvironmentVariable(  "Path",  [Environment]::GetEnvironmentVariable("Path", "Machine") + ";$lo",  "Machine" ) 
```

1. 重新打开终端，验证：

```
where soffice soffice --version 
```

如果只想让当前终端临时生效：

```
$env:Path += ";C:\Program Files\LibreOffice\program" soffice --version 
```

如果 SafeGuard 是以 Windows 服务方式运行，改完系统 PATH 后建议重启服务，必要时重启机器。

**Linux 安装**
Ubuntu / Debian：

```
sudo apt update sudo apt install -y libreoffice 
```

服务器无桌面环境时，建议一起装常见字体和 headless 依赖：

```
sudo apt install -y libreoffice fonts-noto-cjk fonts-liberation libgl1 libglib2.0-0 
```

验证：

```
which soffice soffice --version
```

期望：

- 工作区变更来源明确。
- Python 版本与项目依赖兼容。
- 关键依赖已安装。

### 4.2 启动 Server

Windows 开发环境：

```bat
run_server.bat
```

Linux 测试环境：

```bash
sudo systemctl restart safeguard-api
sudo systemctl status safeguard-api --no-pager
```

健康检查：

```bash
curl http://127.0.0.1:8000/api/v1/health
```

期望：

```json
{"status":"ok","version":"2.0.0"}
```

### 4.3 启动 OCR Service

Windows 开发环境：

```bat
run_ocr_service.bat
```

Linux 测试环境：

```bash
sudo systemctl restart safeguard-ocr
sudo systemctl status safeguard-ocr --no-pager
```

健康检查：

```bash
curl http://127.0.0.1:8010/health
```

期望：

- HTTP 状态码为 `200`。
- 返回中包含 `service: ocr`。
- 如果 `model_loaded=false`，不一定失败，首次 OCR 请求可能才加载模型。

### 4.4 检查 gRPC 上传服务

通过管理接口检查：

```bash
curl -H "Authorization: Bearer dev-admin-token" http://127.0.0.1:8000/api/v1/admin/grpc/health
```

期望：

- 返回 `status` 为 `ok` 或可解释的健康状态。
- 地址与实际 `SAFEGUARD_GRPC_UPLOAD_HOST`、`SAFEGUARD_GRPC_UPLOAD_PORT` 一致。

### 4.5 打开 WebUI

浏览器访问：

```text
http://127.0.0.1:8000/
```

期望：

- 显示 Token 登录页面。
- 输入有效 Token 后进入控制台。
- 顶部导航包含：总览、Agent 管理、资产发现、文件监控、敏感档案、检测规则、事件日志、上传会话、任务管理、运行状态、系统配置。

### 4.6 运行已有自动化测试

当前仓库已有资产发现单元测试：

```powershell
python -m unittest tests.test_asset_discovery
```

当前仓库已有 Agent 本地全链路测试脚本：

```powershell
python test_agent.py --clean
```

可选 USB 事件模拟：

```powershell
python test_agent.py --clean --simulate-usb
```

期望：

- 单元测试全部通过。
- `test_agent.py` 输出 JSON 中 `status` 为 `ok`。
- `checks` 中关键项为 `true`，包括注册、心跳、配置同步、文件上传、事件上报、升级报告。

## 5. 测试数据准备

### 5.1 敏感关键字样本

建议准备以下文本样本：

| 文件名 | 内容示例 | 预期 |
| --- | --- | --- |
| `normal.txt` | 普通会议纪要，不包含敏感词 | 非敏感 |
| `keyword_secret.txt` | 本文件为内部资料，包含客户名单和报价 | 敏感 |
| `id_card.txt` | 身份证号 `110101199003071234` | 敏感，正则命中 |
| `phone_email.txt` | 手机号 `13800138000`，邮箱 `test@example.com` | 敏感，正则命中 |
| `credential.txt` | `api_key=sk-test-123456` 或 `private key` | 敏感 |
| `network.txt` | 内网地址 `192.168.1.10`，MAC `00:11:22:33:44:55` | 敏感 |

### 5.2 文件类型样本

必须覆盖：

```text
.txt, .csv, .docx, .xlsx, .pptx, .pdf, .png, .jpg
```

条件允许时覆盖：

```text
.doc, .ppt, .bmp, .jpeg
```

说明：

- `.doc` 和 `.ppt` 依赖 LibreOffice/soffice 转换能力。
- 图片样本应包含可识别文字，例如“内部资料”“客户名单”“13800138000”。
- PDF 应至少包含一种原生文本 PDF 和一种扫描型 PDF。

### 5.3 边界样本

| 类型 | 建议样本 | 预期 |
| --- | --- | --- |
| 空文件 | `empty.txt` | 不崩溃，检测结果可解释 |
| 超大文件 | 大于配置 `max_file_size_mb` | Agent 跳过或 Server 拒绝，日志可解释 |
| 不支持扩展名 | `.zip`、`.exe` | 不进入敏感检测或标记为不支持 |
| 损坏文件 | 损坏的 `.docx` / `.pdf` | 任务失败可记录，WebUI 可查看失败原因 |
| 重名不同路径 | 两个同名文件放在不同目录 | 文件记录和敏感档案路径正确 |
| 同 hash 不同路径 | 同一文件复制到两个路径 | 敏感档案应按 `agent_id + file_path` 追踪 |

```text

POST /api/v1/uploads/{session_id}/complete
```

- 漏传分片时不能完成。

- 完成上传后触发检测任务。
- 文件 hash、文件名、路径、Agent 关联正确。

### 9.4 规则接口

```text
GET    /api/v1/rules
GET    /api/v1/rules/{rule_id}
POST   /api/v1/rules
PUT    /api/v1/rules/{rule_id}
DELETE /api/v1/rules/{rule_id}
```

测试重点：

- `rule_type` 只允许 `keyword`、`ocr`、`llm`。
- `keyword` 规则必须至少配置 `keywords` 或 `regex_patterns`。
- `ocr` 规则必须配置非空 `keywords` 和有效文件类型。
- `llm` 规则必须配置 `prompt_template`，`threshold` 被限制在 `0.0` 到 `1.0`。
- 删除不存在规则返回 `404`。

### 9.5 敏感档案接口

```text
GET /api/v1/sensitive-files

GET /api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/download
GET /api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/download-highlight
GET /api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/download-diff
```

测试重点：

- 分页参数 `page`、`page_size` 生效。
- `agent_id`、`changed_only`、`is_deleted`、`keyword`、`file_type` 过滤正确。
- 不存在的档案或版本返回 `404`。
- 不存在的归档文件返回 `404`。
- 下载响应的文件名和文件类型合理。

## 11. 端到端业务场景

### 11.1 首次部署到发现敏感文件

步骤：

1. 启动 OCR Service。
2. 启动 Server。
3. 打开 WebUI 并登录。

5. 在测试目录放入 `keyword_secret.txt`。

7. 打开文件监控、敏感档案、事件日志、上传会话。

预期：

- Agent 在线。
- 上传会话完成。
- 文件监控中 `keyword_secret.txt` 为敏感。
- 敏感档案生成一条记录。
- 事件日志包含上传或文件创建相关事件。

### 11.2 敏感文件修改溯源

步骤：

1. 准备已建档敏感文件。
2. 修改文件内容，例如新增一行“客户名单第二版”。
3. 等待 Agent 上传。
4. 打开敏感档案详情。

预期：

- 档案版本数增加。
- `modify_count` 增加。
- 新版本可下载原文件。
- 差异文件可下载，内容反映变更。

### 11.3 敏感文件改名和删除

步骤：

1. 将敏感文件从 `A.txt` 改名为 `B.txt`。
2. 等待事件上报。
3. 删除 `B.txt`。
4. 查看敏感档案。

预期：

- 当前路径更新为 `B.txt`。
- 改名事件记录 old/new path。
- 删除后 `is_deleted=true` 或页面显示已删除。
- 历史版本仍可查看。

### 11.4 配置下发闭环

步骤：

1. 在系统配置中新增扫描目录 `C:\SafeGuardTest\sensitive`。
2. 保存配置。
3. 重启或等待 Agent 拉取配置。
4. 在该目录放入敏感文件。

预期：

- Agent 日志显示配置版本更新。
- 新目录文件被扫描并上传。
- WebUI 中可看到新文件和敏感档案。

### 11.5 Agent 升级包下发

步骤：

1. 在系统配置页面上传测试用 `SafeGuardAgent.exe`。
2. 确认返回版本和 checksum。
3. Agent 拉取心跳后收到升级信息。
4. 查看 Agent 日志和升级状态页面。

预期：

- 升级包上传成功。
- Agent 下载升级包。
- 升级报告在运行状态页面可见。
- 非服务模式可记录“已下载但未重启服务”的合理说明。
