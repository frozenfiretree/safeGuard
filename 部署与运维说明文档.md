## 1. 项目概述

SafeGuard 是一个面向内网终端的敏感文件检测与溯源系统，当前采用 `Server + Windows Agent + Web 前端` 的分布式架构。

系统目标是让 Windows 终端上的 Agent 自动扫描、监控本机指定目录中的文件，将文件或事件上报到 Server；Server 负责文件解析、规则命中、OCR 检测、敏感文件归档、版本历史、diff 和高亮文件生成；前端页面用于查看 Agent 状态、上传记录、敏感文件档案、版本变化、OCR 状态和任务失败情况。

当前主流程如下：

1. Windows Agent 启动后读取本地配置，向 Server 注册并获得 `agent_id` 和 token。
2. Agent 定期 heartbeat，Server 返回是否有配置变化、升级信息等。
3. Agent 扫描 `C:\test` 或下发配置中的目录，发现支持类型文件后计算 hash。
4. Agent 通过 gRPC `10.0.80.28:50051` 分块上传文件。
5. Server 将文件落到本地对象存储或 MinIO，触发检测任务。
6. Server 使用解析器读取 `docx/xlsx/pptx/pdf/txt/csv/image` 等内容；图片或扫描型 PDF 会调用 OCR sidecar。
7. 检测命中规则、OCR 规则或 LLM 复判后，文件被标记为敏感。
8. 敏感文件进入 `guard_state` 归档，建立版本快照、事件时间线、diff 文件和高亮产物。
9. Agent 后续监控 rename / modify / delete 事件并批量上报，Server 更新敏感档案时间线。
10. 前端通过 FastAPI 接口读取 Agent、文件、事件、敏感档案和 OCR 健康状态。

当前实际部署采用轻量单机模式：Linux Server 还未启用 PostgreSQL、Redis、MinIO，使用 SQLite、本地文件存储和内存队列；Agent 只部署在 Windows 目标机器。

## 3. 当前系统已有功能说明

### 3.1 Server 的现有功能

Server 当前已有能力：

- Agent 注册：`POST /api/v1/agents/register`，生成 `agent_id`、token 和配置版本。
- Agent heartbeat：`POST /api/v1/agents/{agent_id}/heartbeat`，维护在线状态、刷新 token、返回配置变更和升级信息。
- 配置下发：`GET /api/v1/agents/{agent_id}/config`，支持全局配置和 Agent 级配置合并。
- HTTP 上传接口：`/api/v1/uploads/...`，保留兼容能力。
- gRPC 上传接口：`0.0.0.0:50051`，用于 Agent 分块上传文件。
- 文件去重：以 `file_hash` 作为检测结果复用依据。
- 检测流水线：解析文本、图片块、OCR、规则检测、LLM 复判。
- 敏感标记：将命中文件写入 `files`、`parse_results`、`rule_hits` 等表。
- 敏感档案归档：为敏感文件建立 `TrackedFile`、`TrackedFileVersion`、`TrackedFileEvent`。
- 版本历史：保存最近若干版本快照，记录 initial / modified / renamed / deleted。
- diff 对比：对文本块进行增删改对比，输出 JSON diff。
- 高亮产物：对 DOCX/PDF 尝试生成高亮文件。
- 静态前端托管：`server/webui/index.html` 存在时，访问 `/` 返回前端页面。
- 管理接口：查看 Agent、文件、事件、上传会话、任务失败、OCR 健康、升级报告等。
- OCR sidecar：独立 FastAPI 服务，默认 `127.0.0.1:8010`。

### 3.2 Agent 的现有功能

Agent 当前仅面向 Windows 目标机器。

已有能力：

- 读取配置：优先环境变量，其次 `install_config.json`，最后使用代码默认值。
- 设备标识：基于 Windows MachineGuid、MAC、hostname 生成 fingerprint。
- 注册与 token：向 Server 注册，保存 `agent_id`、token 和配置版本。
- heartbeat：周期上报状态、配置版本、待上传任务数。
- 配置同步：默认会从 Server 拉取 scan/watch 配置。
- 初始扫描：扫描配置目录，计算 hash，建立 baseline。
- 监控事件：使用 watchdog 监听文件创建、修改、移动、删除。
- USB 轮询：周期检查 USB 相关事件并上报。
- 本地任务队列：SQLite `agent.db` 保存待上传和事件任务。
- gRPC 分块上传：默认 3 MiB 分块，channel 上限 64 MiB。
- 前台测试模式：`python test_agent.py` 注册到 Server 但不安装服务、不自启。
- 本地 dry-run 模式：`python test_agent.py --dry-run` 只本地扫描，不注册、不上传。
- Windows 服务安装：`SafeGuardAgent.exe install --server-base ...`。

当前 Agent 不是 Linux Agent；Linux 上部署的是 Server。

## 4. 部署前准备

### 4.1 Linux Server 环境建议

推荐：

- Ubuntu 22.04/24.04 或同类 Linux。
- 建议Python 3.12。
- nginx 1.24 或系统包版本。
- git。
- 可选 Docker / Docker Compose。当前轻量模式不使用 PostgreSQL、Redis、MinIO。
- OCR 需要 `libgl1`、`libglib2.0-0` 等图像运行库。

安装系统依赖：

```bash
sudo apt update
sudo apt install -y git nginx libreoffice libgl1 libglib2.0-0 unzip
```

如果后续使用 Docker 基础设施：

```bash
sudo apt install -y docker.io docker-compose-plugin
```

### 4.2 Python 环境

建议：

```bash
conda create -y -n scanocr python=3.12
conda activate scanocr
cd /opt/safeguard
pip install -r requirements.txt
```

当前 RTX 5090 的 Paddle GPU wheel 存在架构不匹配问题，生产先用 CPU OCR 更稳：

```bash
conda activate scanocr
python -m pip uninstall -y paddlepaddle-gpu paddlepaddle
python -m pip install paddlepaddle
```

### 4.3 端口要求

| 端口 | 用途 | 暴露建议 |
| --- | --- | --- |
| `80/tcp` | nginx 前端/API 入口 | 对 Windows 管理机和 Agent 可访问 |
| `8000/tcp` | FastAPI 后端 | 仅本机监听或内网访问，nginx 反代 |
| `8010/tcp` | OCR sidecar | 只监听 `127.0.0.1` |
| `50051/tcp` | gRPC 文件上传 | Agent 必须可访问 |
| `5432/6379/9000` | PostgreSQL/Redis/MinIO | 当前轻量模式不使用 |

Linux 防火墙：

```bash
sudo ufw allow 80/tcp
sudo ufw allow 50051/tcp
```

### 4.4 模型文件

OCR 依赖本地模型目录：

```text
/opt/safeguard/server/artifacts/models/paddlex_models
```

应包含：

```text
PP-LCNet_x1_0_doc_ori
PP-LCNet_x1_0_textline_ori
PP-OCRv5_server_det
PP-OCRv5_server_rec
UVDoc
```

如果只有 `models.zip`：

```bash
cd /opt/safeguard/server/artifacts
unzip -o models.zip -d models
```

## 5. Server 部署步骤

### 5.1 获取代码

首次部署：

```bash
cd /opt
sudo git clone https://github.com/frozenfiretree/safeGuard.git safeguard
sudo chown -R user:user /opt/safeguard
```

更新代码：

```bash
cd /opt/safeguard
git pull
```

如果出现权限错误：

```bash
sudo chown -R user:user /opt/safeguard
sudo chmod -R u+rwX /opt/safeguard
git pull
```

### 5.2 安装依赖

```bash
conda activate scanocr
cd /opt/safeguard
pip install -r requirements.txt
```

### 5.3 配置轻量模式环境变量

创建配置：

```bash
sudo mkdir -p /etc/safeguard
sudo nano /etc/safeguard/server.env
```

示例：

```env
SAFEGUARD_REQUIRE_PRODUCTION_DEPS=false

SAFEGUARD_DATABASE_URL=sqlite:////opt/safeguard/server/data/server_v2.db
SAFEGUARD_CELERY_BROKER_URL=memory://
SAFEGUARD_CELERY_RESULT_BACKEND=cache+memory://
SAFEGUARD_REDIS_URL=

SAFEGUARD_MINIO_ENDPOINT=
SAFEGUARD_MINIO_ACCESS_KEY=
SAFEGUARD_MINIO_SECRET_KEY=
SAFEGUARD_MINIO_SECURE=false
SAFEGUARD_MINIO_BUCKET=safeguard

SAFEGUARD_OCR_SERVICE_URL=http://127.0.0.1:8010
SAFEGUARD_OCR_SERVICE_TIMEOUT_SECONDS=60
SAFEGUARD_OCR_USE_GPU=false
SAFEGUARD_OCR_GPU_ID=0

SAFEGUARD_GRPC_UPLOAD_HOST=0.0.0.0
SAFEGUARD_GRPC_UPLOAD_PORT=50051

SAFEGUARD_ADMIN_TOKEN=replace-with-ascii-random-token（当前默认admin）
SAFEGUARD_ADMIN_USER=admin
SAFEGUARD_ADMIN_PASSWORD=replace-with-ascii-password（当前默认admin）
```

生成 token：

```bash
openssl rand -hex 32
```

不要使用中文 token，否则浏览器 fetch 设置 header 时会报：

```text
String contains non ISO-8859-1 code point
```

```bash
cd /opt/safeguard/server

mkdir -p data/tmp_uploads_v1 data/assets logs
sudo chown -R user:user /opt/safeguard/server/data /opt/safeguard/server/logs /opt/safeguard/server/artifacts
```

```bash

```

### 5.6 nginx

配置：

```bash
sudo nano /etc/nginx/conf.d/safeguard.conf
```

示例：

```nginx
upstream safeguard_api {
    server 127.0.0.1:8000;
}

server {
    listen 80;
    server_name 10.0.80.28;
    client_max_body_size 100m;

    location / {
        proxy_pass http://safeguard_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 600s;
    }
}
```

如果默认站点冲突：

```bash
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -T | grep -nE "server_name|listen 80|root |proxy_pass"
```

启动：

```bash
sudo nginx -t
sudo systemctl restart nginx
sudo systemctl enable nginx
```

## 6. Agent 部署与运行步骤

将 `SafeGuardAgent.exe` 放到目标 Windows 机器。推荐同目录放置：

```text
SafeGuardAgent.exe

```

```json
{
  "server_base": "http://10.0.80.28",
  "work_dir": "C:\\ProgramData\\SafeGuardAgent"

```

```powershell
.\SafeGuardAgent.exe remove
Remove-Item -Force "C:\ProgramData\SafeGuardAgent\install_config.json" -ErrorAction SilentlyContinue
.\SafeGuardAgent.exe install --server-base "http://10.0.80.28" --work-dir "C:\ProgramData\SafeGuardAgent"
Start-Service SafeGuardAgent
```

检查：

```powershell
Get-Service SafeGuardAgent
Get-Content "C:\ProgramData\SafeGuardAgent\install_config.json"
Get-Content "C:\ProgramData\SafeGuardAgent\logs\agent.log" -Tail 200
```

适合开发机临时测试：注册到服务器、heartbeat、上传，但不安装服务、不自启。

```powershell
cd D:\work\safeGuard
```

## 7. 使用方法

### 7.1 登录/访问前端

访问：

```text
http://10.0.80.28/
```

管理接口需要 `SAFEGUARD_ADMIN_TOKEN` 或 Basic Auth。前端如果要求 token，填 `/etc/safeguard/server.env` 中的 ASCII token（当前默认admin）。

### 7.2 注册 Agent

正式服务安装或 `test_agent.py` 前台模式启动后会自动注册。前端 Agent 列表应出现 hostname、IP、状态、配置版本。

### 7.3 上传扫描结果

Agent 扫描支持扩展名：

```text
.docx .xlsx .pdf .pptx .csv .txt .png .jpg .jpeg .bmp
```

上传流程：

1. Agent 入队 `UPLOAD`。
2. gRPC `InitUpload` 建立上传会话。
3. Agent 按 chunk 发送文件。
4. Server 合并 chunk，校验 sha256。
5. Server 触发检测任务。

### 7.4 生成敏感文件档案

只要检测命中规则/OCR/LLM 结果，Server 将 `FileRecord.is_sensitive=True` 并进入敏感档案归档。当前已修复同 hash 文件 dedup 后不归档的问题：相同内容但不同路径的敏感文件会按路径建立档案。

### 7.5 修改、重命名、删除事件

Agent 监听 watch_dirs：

- 修改：上报 `file_modified`，Server 建立新版本。
- 重命名/移动：上报 rename/move，Server 更新时间线。
- 删除：上报 delete，Server 标记已删除。

### 7.6 下载版本、高亮、diff

前端敏感档案详情页可下载：

- 原始版本文件。
- DOCX/PDF 高亮文件。
- diff JSON。

后端接口路径形态：

```text
/api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/download
/api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/highlight
/api/v1/sensitive-files/{tracked_file_id}/versions/{version_id}/diff
```

## 8. 测试手册

### 8.1 基础连通性测试

前置条件：

- `safeguard-api`、`safeguard-ocr`、`nginx` 已启动。
- Windows 能访问 `10.0.80.28`。

操作：

## 9. 部署过程中的坑与规避指南

### 9.1 nginx 502 Bad Gateway

现象：Windows 访问 `http://10.0.80.28/` 显示 502。

原因：nginx 到 `127.0.0.1:8000` 的 API 后端不通。

解决：

```bash
curl -v http://127.0.0.1:8000/
sudo systemctl status safeguard-api --no-pager
sudo journalctl -u safeguard-api -n 120 --no-pager
```

避免：每次改 systemd 后先本机 curl 8000，再测 nginx。

### 9.2 systemd `203/EXEC`

现象：

```text
status=203/EXEC
Failed to execute /opt/conda/envs/scanocr/bin/uvicorn
```

原因：ExecStart 路径不存在或无权限。

解决：使用真实 Python 路径，并用 `python -m uvicorn`：

```ini
ExecStart=/home/user/anaconda3/envs/scanocr/bin/python -m uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1
```

避免：不要假设 conda 在 `/opt/conda`，先 `which python`。

### 9.3 `/opt/safeguard` 权限错误

现象：

```text
PermissionError: /opt/safeguard/server/data/object_store
git pull unable to unlink old '.gitignore'
```

原因：目录归属 root 或其他用户。

解决：

```bash
sudo chown -R user:user /opt/safeguard
sudo chmod -R u+rwX /opt/safeguard
```

避免：systemd `User=` 与目录 owner 保持一致。

### 9.4 OCR 尝试联网下载模型

现象：

```text
download model from huggingface
Network is unreachable
```

原因：PaddleOCR 没有使用项目本地模型，或某些预处理模型未显式指定。

解决：当前代码已显式指定本地模型目录；更新代码后重启：

```bash
cd /opt/safeguard
git pull
sudo systemctl restart safeguard-ocr
```

避免：部署前检查 `server/artifacts/models/paddlex_models`。

### 9.5 RTX 5090 GPU OCR 不兼容

现象：

```text
Mismatched GPU Architecture
current GPU is 120
compiled for 61 70 75 80 86 89 90
```

原因：当前 PaddlePaddle GPU wheel 不支持 `sm_120`。

解决：先切 CPU：

```env
SAFEGUARD_OCR_USE_GPU=false
```

```bash
python -m pip uninstall -y paddlepaddle-gpu paddlepaddle
python -m pip install paddlepaddle
sudo systemctl restart safeguard-ocr
```

避免：5090 上 GPU OCR 需要等待支持 Blackwell 的 Paddle wheel，或使用支持 `sm_120` 的 Docker/源码编译版本。

### 9.6 Admin Token 使用中文导致前端 fetch 报错

现象：

```text
String contains non ISO-8859-1 code point
```

原因：`SAFEGUARD_ADMIN_TOKEN=换成强随机token` 被当作 Header 值。

解决：

```bash
openssl rand -hex 32
sudo nano /etc/safeguard/server.env
sudo systemctl restart safeguard-api
```

避免：token/password 使用 ASCII。

## 10. 当前系统不足

- 轻量模式适合单机测试和小规模部署，不适合大规模并发。SQLite、memory Celery、本地对象存储没有横向扩展能力。
- API 与 gRPC 当前在同一个 FastAPI 进程里启动，因此 API systemd 必须 `--workers 1`。要多 worker 需要拆分 gRPC 服务。
- OCR 对 GPU 环境敏感，RTX 5090 暂不适合直接使用当前 Paddle GPU wheel。
- OCR/PDF 高亮仍有边界：扫描型 PDF 无原生文本坐标时，高亮只能有限回退。
- Agent 打包和安装链路仍需人工确认 server IP、权限、UAC 和环境变量残留。
- rename / save-as / delete 语义仍有边界，尤其不同程序保存文件时可能表现为 delete + create。
- 敏感档案 backfill 依赖页面或接口触发，后台定时补偿在轻量模式下较弱。
- 前端主要是管理视图，告警、筛选、批量操作和审计能力还可增强。
- 日志可观测性还不够，缺少统一 trace_id、文件 hash 生命周期追踪和结构化错误面板。

## 12. 性能优化与工程改进建议

### 短期优化

- OCR 先 CPU 稳定运行，GPU 等 Paddle 支持 `sm_120` 后再开启。
- 大文件上传前做大小过滤，减少无意义上传。
- Server 日志增加 `file_hash`、`agent_id`、`file_path` 关键字段。
- 前端轮询频率可按页面可见性降低，减少后台请求。

### 中期优化

- 引入 Redis + Celery worker，将检测任务从 API 进程中拆出。
- 文件对象存储迁移到 MinIO，避免本地磁盘单点和迁移困难。
- 对 PDF 解析和 OCR 按页并发，限制最大并发避免内存爆。
- diff 计算可存摘要和分页详情，避免大文档 diff JSON 过大。
- 敏感档案版本保留策略可配置，例如只保留最近 3 个快照，长期版本只保留元数据。

### 长期优化

- PostgreSQL 作为主数据库，支持更多 Agent 和更复杂查询。
- gRPC 独立服务 + API 多 worker + worker 队列，形成清晰服务边界。
- Agent 支持更可靠的文件身份策略，结合 USN Journal 或文件 ID，减少 rename/save-as 边界误判。
- 建立完整监控：服务健康、队列长度、OCR 耗时、上传失败率、磁盘使用率。
- 建立 CI：单元测试、集成测试、PyInstaller 打包验证、Linux systemd 模板验证。

## 13. FAQ / 故障排查速查表

### Q: 前端 502 怎么办？

```bash
curl http://127.0.0.1:8000/
sudo systemctl status safeguard-api --no-pager
sudo journalctl -u safeguard-api -n 120 --no-pager
```

先修 API，再修 nginx。

### Q: Agent 无法注册怎么办？

Windows：

```powershell
Get-Content "C:\ProgramData\SafeGuardAgent\install_config.json"
Get-Content "C:\ProgramData\SafeGuardAgent\logs\agent.log" -Tail 200
curl http://10.0.80.28/api/v1/health
```

Linux：

```bash
tail -f /opt/safeguard/server/logs/server_v2.log
```

应看到 `POST /api/v1/agents/register`。

### Q: Agent 注册了但上传失败？

检查 gRPC：

```powershell
Test-NetConnection 10.0.80.28 -Port 50051
```

Linux：

```bash
sudo ss -lntp | grep 50051
sudo ufw allow 50051/tcp
```

### Q: OCR error 怎么办？

```bash
sudo systemctl status safeguard-ocr --no-pager
sudo journalctl -u safeguard-ocr -n 120 --no-pager
curl -s http://127.0.0.1:8010/health
```

看 `init_error`、`model_home`、`paddle_cuda`。

### Q: 页面能打开但 OCR 显示 error？

API 到 OCR 不通或 OCR 初始化失败：

```bash
curl http://127.0.0.1:8010/health
sudo systemctl restart safeguard-ocr
```

### Q: 清空测试数据怎么做？

```bash
sudo systemctl stop safeguard-api safeguard-ocr nginx
cd /opt/safeguard/server
rm -f data/server_v2.db
rm -rf data/object_store data/tmp_uploads_v1 data/guard_state data/assets
mkdir -p data/object_store/uploads data/object_store/files data/object_store/upgrades data/tmp_uploads_v1 data/assets
sudo chown -R user:user data logs
sudo systemctl start safeguard-ocr safeguard-api nginx
```

Windows 测试 Agent：

```powershell
Remove-Item -Recurse -Force .\.agent_local_dryrun
```

### Q: `git pull` 权限不足怎么办？

```bash
cd /opt/safeguard
sudo chown -R user:user /opt/safeguard
sudo chmod -R u+rwX /opt/safeguard
git pull
```
