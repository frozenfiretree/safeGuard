## 1. 文档目标与适用范围

本文档用于指导后续维护者在一台完全陌生的 Linux 服务器上，从 GitHub 拉取 SafeGuard 项目代码，配置运行环境，启动 Server、OCR、前端页面和 Windows Agent，并完成端到端验证。

本文档按当前真实项目状态编写，而不是通用 Python 项目模板。当前推荐部署方式是：

- Server 部署在 Linux 服务器。
- Agent 只部署在 Windows 目标机器。
- Server 采用轻量单机模式，不启用 PostgreSQL、Redis、MinIO。
- 数据库使用 SQLite。
- 队列使用 Celery memory 模式。
- 对象存储使用本地目录。
- 前端由 FastAPI 托管 `server/webui`，nginx 负责反向代理。
- OCR 作为独立 sidecar 服务运行。

当前已验证过的示例服务器 IP 是：

```text
10.0.80.28
```

如果部署到其他服务器，请将文档中的 `10.0.80.28` 替换成实际服务器 IP 或域名。

## 2. 系统架构与部署拓扑

### 2.1 组件说明

| 组件 | 运行位置 | 作用 |
| --- | --- | --- |
| FastAPI Server | Linux | 接收 Agent 注册、心跳、上传文件、事件上报，提供管理 API 和前端静态页面。 |
| gRPC Upload Server | Linux，随 FastAPI API 进程启动 | 接收 Agent 分块上传文件，默认端口 `50051`。 |
| OCR Service | Linux | 独立 FastAPI 服务，默认监听 `127.0.0.1:8010`，供 Server 调用。 |
| Web 前端 | Linux，由 FastAPI 托管 | 位于 `server/webui`，通过 nginx 对外访问。 |
| Windows Agent | Windows 目标机 | 扫描文件、监听变化、注册到 Server、上传文件和事件。 |
| SQLite | Linux 本地文件 | 轻量模式数据库，路径 `server/data/server_v2.db`。 |
| 本地对象存储 | Linux 本地目录 | 保存上传文件、临时 chunk、敏感文件版本快照。 |

### 2.2 端口规划

| 端口 | 监听地址 | 用途 | 是否需要对 Windows Agent 开放 |
| --- | --- | --- | --- |
| `80` | `0.0.0.0` | nginx 对外入口，前端和 HTTP API | 需要 |
| `8000` | `0.0.0.0` 或 `127.0.0.1` | FastAPI API | 通常不直接开放，由 nginx 代理 |
| `8010` | `127.0.0.1` | OCR sidecar | 不需要 |
| `50051` | `0.0.0.0` | gRPC 文件上传 | 需要 |

### 2.3 数据流

1. Windows Agent 启动。
2. Agent 读取 `server_base`，例如 `http://10.0.80.28`。
3. Agent 调用 `POST /api/v1/agents/register` 注册。
4. Agent 调用 heartbeat 获取配置变化。
5. Agent 扫描本地文件，计算 sha256。
6. Agent 通过 gRPC `10.0.80.28:50051` 上传文件。
7. Server 合并 chunk，写入本地对象存储。
8. Server 解析文件并调用 OCR。
9. 命中规则后，Server 写入敏感文件档案。
10. 前端展示 Agent、文件、事件、敏感档案、OCR 健康状态。

## 3. 从零部署总览

完整部署顺序如下：

1. 登录 Linux 服务器。
2. 安装系统依赖：git、nginx、libGL、LibreOffice 等。
3. 准备 conda 或 venv Python 环境。
4. 从 GitHub clone 项目到 `/opt/safeguard`。
5. 安装 Python 依赖。
6. 创建 `/etc/safeguard/server.env`。
7. 检查 `server/webui` 前端文件是否存在。
8. 检查 OCR 模型目录是否存在。
9. 初始化 `server/data` 和 `server/logs`。
10. 写入 `safeguard-api.service`。
11. 写入 `safeguard-ocr.service`。
12. 配置 nginx。
13. 启动 `safeguard-ocr`、`safeguard-api`、`nginx`。
14. 使用 curl 验证 API、OCR、前端、gRPC 端口。
15. 在 Windows 机器运行 Agent。
16. 在前端确认 Agent 注册、文件上传、敏感档案生成。

## 4. 新服务器基础环境准备

### 4.1 确认系统信息

登录服务器后执行：

```bash
whoami
pwd
uname -a
cat /etc/os-release
ip addr
```

确认你知道以下信息：

- 当前登录用户，例如 `user`。
- 服务器 IP，例如 `10.0.80.28`。
- 是否具备 `sudo` 权限。
- 是否能访问 GitHub。

检查 sudo：

```bash
sudo -v
```

如果没有 sudo 权限，后续无法安装系统依赖、写 systemd、配置 nginx。

### 4.2 安装系统依赖

Ubuntu/Debian：

```bash
sudo apt update
sudo apt install -y git nginx unzip curl wget libreoffice libgl1 libglib2.0-0
```

说明：

- `git`：从 GitHub 拉取代码。
- `nginx`：提供前端和 API 统一入口。
- `unzip`：解压 OCR 模型包。
- `libreoffice`：用于 `.doc` 转 `.docx` 等文档转换能力。
- `libgl1`、`libglib2.0-0`：OpenCV/PaddleOCR 常见运行库。

检查 nginx：

```bash
nginx -v
sudo systemctl status nginx --no-pager
```

nginx 没启动不是问题，后面会统一配置。

### 4.3 防火墙与安全组

Linux 本机使用 ufw 时：

```bash
sudo ufw allow 80/tcp
sudo ufw allow 50051/tcp
sudo ufw status
```

如果服务器在云平台，还要在云安全组放行：

- TCP 80
- TCP 50051

`8010` 不要对外开放，OCR 只给本机 API 调用。

### 4.4 磁盘空间

查看磁盘：

```bash
df -h
```

建议至少预留：

- 代码和依赖：5 GB。
- OCR 模型：数 GB。
- 上传文件和敏感档案：按业务规模预估，测试环境至少 20 GB。

## 5. 从 GitHub 拉取项目代码

### 5.1 首次 clone

推荐部署目录：

```text
/opt/safeguard
```

执行：

```bash
cd /opt
sudo git clone https://github.com/frozenfiretree/safeGuard.git safeguard
sudo chown -R user:user /opt/safeguard
cd /opt/safeguard
git status
```

把 `user:user` 替换为当前实际运行服务的用户。本文档后续假设运行用户是 `user`。

如果使用私有仓库：

```bash
git clone https://<token>@github.com/<owner>/<repo>.git safeguard
```

或配置 SSH key 后：

```bash
git clone git@github.com:frozenfiretree/safeGuard.git safeguard
```

### 5.2 如果 `/opt/safeguard` 已存在

先确认它是不是 Git 仓库：

```bash
cd /opt/safeguard
git status
```

如果不是 Git 仓库，不要直接覆盖。先备份：

```bash
cd /opt
sudo mv safeguard safeguard.bak.$(date +%Y%m%d%H%M%S)
sudo git clone https://github.com/frozenfiretree/safeGuard.git safeguard
sudo chown -R user:user /opt/safeguard
```

如果旧目录有运行数据，需要迁回：

```bash
sudo cp -a /opt/safeguard.bak.*/server/data /opt/safeguard/server/ 2>/dev/null || true
sudo cp -a /opt/safeguard.bak.*/server/logs /opt/safeguard/server/ 2>/dev/null || true
sudo chown -R user:user /opt/safeguard/server/data /opt/safeguard/server/logs
```

### 5.3 Git 权限问题处理

如果 `git pull` 报：

```text
unable to unlink old '.gitignore': Permission denied
unable to create file test_agent.py: Permission denied
```

执行：

```bash
cd /opt/safeguard
sudo chown -R user:user /opt/safeguard
sudo chmod -R u+rwX /opt/safeguard
git pull
```

如果 Windows 本地开发目录出现 `dubious ownership`：

```powershell
git config --global --add safe.directory D:/work/safeGuard
```

## 6. 创建 Python 运行环境

### 6.1 推荐使用 conda

如果服务器已经安装 Anaconda/Miniconda：

```bash
conda create -y -n scanocr python=3.12
conda activate scanocr
python --version
```

安装依赖：

```bash
cd /opt/safeguard
pip install --upgrade pip
pip install -r requirements.txt
```

确认关键依赖：

```bash
python -c "import fastapi, uvicorn, grpc, sqlalchemy; print('basic deps ok')"
python -c "import paddleocr; print('paddleocr ok')"
```

### 6.2 如果使用 venv

```bash
sudo apt install -y python3.12-venv python3-pip
cd /opt/safeguard
python3.12 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

systemd 里的 Python 路径要改成：

```text
/opt/safeguard/.venv/bin/python
```

### 6.3 OCR 与 PaddlePaddle 说明

当前 RTX 5090 GPU 会遇到 PaddlePaddle wheel 不支持 `sm_120` 的问题：

```text
Mismatched GPU Architecture
current GPU is 120
compiled for 61 70 75 80 86 89 90
```

新服务器部署时，建议先使用 CPU OCR：

```bash
conda activate scanocr
python -m pip uninstall -y paddlepaddle-gpu paddlepaddle
python -m pip install paddlepaddle
```

不要急着启用 GPU。先确保系统完整跑通。

## 7. 配置 Server 运行参数

### 7.1 创建 `/etc/safeguard/server.env`

```bash
sudo mkdir -p /etc/safeguard
sudo nano /etc/safeguard/server.env
```

写入以下内容。注意将 token 和密码替换成真实 ASCII 字符串。

```env
SAFEGUARD_REQUIRE_PRODUCTION_DEPS=false

SAFEGUARD_DATABASE_URL=sqlite:////opt/safeguard/server/data/server_v2.db
SAFEGUARD_CELERY_BROKER_URL=memory://
SAFEGUARD_CELERY_RESULT_BACKEND=cache+memory://
SAFEGUARD_REDIS_URL=

SAFEGUARD_MINIO_ENDPOINT=
SAFEGUARD_MINIO_ACCESS_KEY=
SAFEGUARD_MINIO_SECRET_KEY=
SAFEGUARD_MINIO_SECURE=false
SAFEGUARD_MINIO_BUCKET=safeguard

SAFEGUARD_OCR_SERVICE_URL=http://127.0.0.1:8010
SAFEGUARD_OCR_SERVICE_TIMEOUT_SECONDS=60
SAFEGUARD_OCR_USE_GPU=false
SAFEGUARD_OCR_GPU_ID=0

SAFEGUARD_GRPC_UPLOAD_HOST=0.0.0.0
SAFEGUARD_GRPC_UPLOAD_PORT=50051

SAFEGUARD_ADMIN_TOKEN=replace-with-ascii-random-token
SAFEGUARD_ADMIN_USER=admin
SAFEGUARD_ADMIN_PASSWORD=replace-with-ascii-password
```

生成 token：

```bash
openssl rand -hex 32
```

示例：

```env
SAFEGUARD_ADMIN_TOKEN=2c8d59d00f0b4e3f8a0a5d5c7b7e1a5ddf3cf4d4ad0b8af391e41b88dd90abcd
```

不要使用中文占位值，例如：

```env
SAFEGUARD_ADMIN_TOKEN=换成强随机token
```

否则前端调用管理接口时会报：

```text
Failed to read the 'headers' property from 'RequestInit':
String contains non ISO-8859-1 code point.
```

### 7.2 确认配置文件权限

```bash
sudo chmod 600 /etc/safeguard/server.env
sudo chown root:root /etc/safeguard/server.env
```

systemd 服务可以读取 root-owned EnvironmentFile。

### 7.3 轻量模式与生产模式差异

当前文档默认轻量模式：

- SQLite：`server/data/server_v2.db`
- Redis：不使用，代码回退到内存缓存。
- MinIO：不使用，代码回退到本地对象存储。
- Celery：`memory://`，任务在当前进程环境中执行。
- worker/beat：不启用。

生产模式未来可启用：

- PostgreSQL
- Redis
- MinIO
- Celery worker
- Celery beat

但当前项目实际部署和排障以轻量模式为准。

## 8. 准备数据目录、模型目录和前端文件

### 8.1 初始化 Server 数据目录

```bash
cd /opt/safeguard/server
mkdir -p data/object_store/uploads
mkdir -p data/object_store/files
mkdir -p data/object_store/upgrades
mkdir -p data/tmp_uploads_v1
mkdir -p data/assets
mkdir -p logs

```

首次启动后会自动生成：

```text
server/data/server_v2.db
server/logs/server_v2.log
```

`server/data/guard_state` 只有在敏感文件归档后才会出现。没有该目录不代表部署失败。

### 8.2 检查 OCR 模型目录

执行：

```bash
ls -lah /opt/safeguard/server/artifacts
ls -lah /opt/safeguard/server/artifacts/models
ls -lah /opt/safeguard/server/artifacts/models/paddlex_models
```

应看到：

```text
PP-LCNet_x1_0_doc_ori
PP-LCNet_x1_0_textline_ori
PP-OCRv5_server_det
PP-OCRv5_server_rec
UVDoc
```

如果只有：

```text
/opt/safeguard/server/artifacts/models.zip
```

则解压：

```bash
cd /opt/safeguard/server/artifacts
unzip -o models.zip -d models
sudo chown -R user:user /opt/safeguard/server/artifacts
```

### 8.3 检查前端静态文件

FastAPI 会从这里托管前端：

```text
/opt/safeguard/server/webui/index.html
/opt/safeguard/server/webui/assets/
```

检查：

```bash
ls -lah /opt/safeguard/server/webui
ls -lah /opt/safeguard/server/webui/assets
```

如果 `server/webui/index.html` 不存在，访问 `/` 只会返回：

```json
{"status":"ok","version":"2.0.0"}
```

这种情况下需要在开发机先构建前端，再把构建产物复制到 `server/webui`。如果仓库已经包含 `server/webui`，直接确认文件存在即可。

## 9. 配置 systemd 服务

### 9.1 确认 Python 路径

```bash
conda activate scanocr
which python
python -c "import sys; print(sys.executable)"
```

示例输出：

```text
/home/user/anaconda3/envs/scanocr/bin/python
```

后续 systemd 中必须使用这个真实路径。不要盲目使用 `/opt/conda/envs/scanocr/bin/python`。

### 9.2 API 服务

创建文件：

```bash
sudo nano /etc/systemd/system/safeguard-api.service
```

内容：

```ini
[Unit]
Description=SafeGuard API Service
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/safeguard/server
EnvironmentFile=/etc/safeguard/server.env
ExecStart=/home/user/anaconda3/envs/scanocr/bin/python -m uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1
Restart=always
RestartSec=5
User=user
Group=user

[Install]
WantedBy=multi-user.target
```

注意：

- `ExecStart` 改成你的真实 Python 路径。
- `--workers 1` 必须保留。当前 gRPC upload server 在 FastAPI startup 中启动，多 worker 会抢占 `50051`。
- `User=user` 要与 `/opt/safeguard` 目录权限一致。

### 9.3 OCR 服务

创建文件：

```bash
sudo nano /etc/systemd/system/safeguard-ocr.service
```

内容：

```ini
[Unit]
Description=SafeGuard OCR Service
After=network.target

[Service]
Type=simple
WorkingDirectory=/opt/safeguard/server
EnvironmentFile=/etc/safeguard/server.env
ExecStart=/home/user/anaconda3/envs/scanocr/bin/python -m uvicorn ocr_service:app --host 127.0.0.1 --port 8010 --workers 1
Restart=always
RestartSec=5
User=user
Group=user

[Install]
WantedBy=multi-user.target
```

注意：

- OCR 只监听 `127.0.0.1`。
- 如果原来用 `User=scanocr`，但 Python 环境在 `/home/user/anaconda3/...`，会报 Permission denied。要么把环境迁到 `/opt`，要么用 `User=user`。

### 9.4 禁用当前不使用的服务

轻量模式不启用 worker/beat：

```bash
sudo systemctl disable --now safeguard-worker safeguard-beat 2>/dev/null || true
```

### 9.5 重新加载 systemd

```bash
sudo systemctl daemon-reload
sudo systemctl enable safeguard-ocr safeguard-api
```

先不要急着启动 nginx，先启动后端和 OCR：

```bash
sudo systemctl start safeguard-ocr
sudo systemctl start safeguard-api
```

检查：

```bash
sudo systemctl status safeguard-ocr --no-pager
sudo systemctl status safeguard-api --no-pager
```

## 10. 配置 nginx 反向代理

### 10.1 写入 SafeGuard nginx 配置

```bash
sudo nano /etc/nginx/conf.d/safeguard.conf
```

内容：

```nginx
upstream safeguard_api {
    server 127.0.0.1:8000;
}

server {
    listen 80;
    server_name 10.0.80.28;
    client_max_body_size 100m;

    location / {
        proxy_pass http://safeguard_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 600s;
    }
}
```

将 `10.0.80.28` 替换为实际服务器 IP 或域名。

### 10.2 处理默认站点冲突

查看 nginx 配置：

```bash
sudo nginx -T | grep -nE "server_name|listen 80|root |proxy_pass"
```

如果有默认站点或旧站点占用 80，例如：

```text
server_name _;
root /var/www/cps-platform;
```

禁用默认站点：

```bash
sudo rm -f /etc/nginx/sites-enabled/default
```

如果旧配置在 `/etc/nginx/conf.d/xxx.conf`，移动到禁用目录：

```bash
sudo mkdir -p /etc/nginx/disabled
sudo mv /etc/nginx/conf.d/xxx.conf /etc/nginx/disabled/
```

### 10.3 启动 nginx

```bash
sudo nginx -t
sudo systemctl restart nginx
sudo systemctl enable nginx
```

如果提示：

```text
nginx.service is not active, cannot reload
```

说明 nginx 没启动，执行：

```bash
sudo systemctl start nginx
```

## 11. 启动服务并做最小验收

### 11.1 启动顺序

```bash
sudo systemctl restart safeguard-ocr
sudo systemctl restart safeguard-api
sudo systemctl restart nginx
```

### 11.2 检查监听端口

```bash
sudo ss -lntp | grep -E "80|8000|8010|50051"
```

期望看到：

```text
0.0.0.0:80
0.0.0.0:8000
127.0.0.1:8010
0.0.0.0:50051
```

### 11.3 检查 API

```bash
curl -v http://127.0.0.1:8000/api/v1/health
```

期望：

```json
{"status":"ok","version":"2.0.0"}
```

### 11.4 检查 OCR

```bash
curl -s http://127.0.0.1:8010/health
```

期望：

```json
{
  "status": "ok",
  "service": "ocr",
  "ready": true
}
```

如果 `model_loaded=false`，不一定是错误，表示还没有真正执行 OCR 请求。第一次 OCR 请求才会加载模型。

### 11.5 检查前端

```bash
curl -v http://10.0.80.28/
```

如果返回 HTML，说明前端正常。

如果返回：

```json
{"status":"ok","version":"2.0.0"}
```

说明前端静态文件缺失，检查：

```bash
ls -lah /opt/safeguard/server/webui/index.html
```

### 11.6 检查管理接口

```bash
ADMIN_TOKEN="你的SAFEGUARD_ADMIN_TOKEN"
curl -H "Authorization: Bearer $ADMIN_TOKEN" http://127.0.0.1:8000/api/v1/admin/agents
curl -H "Authorization: Bearer $ADMIN_TOKEN" http://127.0.0.1:8000/api/v1/admin/ocr/health
```

如果 401，检查 token 是否与 `/etc/safeguard/server.env` 一致，并重启 API：

```bash
sudo systemctl restart safeguard-api
```

## 12. Windows Agent 部署与测试

### 12.1 正式安装 Agent

在 Windows 目标机上准备：

```text
SafeGuardAgent.exe
agent-install.json
```

`agent-install.json`：

```json
{
  "server_base": "http://10.0.80.28",
  "work_dir": "C:\\ProgramData\\SafeGuardAgent"
}
```

## 14. 常见故障与准确处理方法

### 14.1 前端 502 Bad Gateway

现象：

```text
502 Bad Gateway
```

原因：nginx 访问不到 `127.0.0.1:8000`。

处理：

```bash
curl -v http://127.0.0.1:8000/
sudo systemctl status safeguard-api --no-pager
sudo journalctl -u safeguard-api -n 120 --no-pager
```

先让 `curl http://127.0.0.1:8000/` 成功，再重启 nginx。

### 14.2 systemd `203/EXEC`

现象：

```text
status=203/EXEC
Failed to execute .../uvicorn
Permission denied
```

原因：

- `ExecStart` 路径错误。
- systemd 用户没有权限进入 Python 环境目录。
- 直接执行 `uvicorn` 脚本权限有问题。

处理：

```bash
which python
```

使用：

```ini
ExecStart=/home/user/anaconda3/envs/scanocr/bin/python -m uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1
```

不要直接写不存在的：

```text
/opt/conda/envs/scanocr/bin/uvicorn
```

### 14.3 目录权限导致服务启动失败

现象：

```text
PermissionError: /opt/safeguard/server/data/object_store
```

处理：

```bash
sudo chown -R user:user /opt/safeguard
sudo chmod -R u+rwX /opt/safeguard
sudo systemctl restart safeguard-api safeguard-ocr
```

### 14.4 OCR 尝试从 HuggingFace 下载模型

现象：

```text
Network is unreachable
Using official model ... automatically downloaded
```

原因：本地模型目录缺失，或 PaddleOCR 没使用本地模型。

处理：

```bash
ls /opt/safeguard/server/artifacts/models/paddlex_models
cd /opt/safeguard
git pull
sudo systemctl restart safeguard-ocr
```

### 14.5 RTX 5090 GPU OCR 报 `Unsupported GPU architecture`

原因：当前 Paddle GPU wheel 不支持 `sm_120`。

处理：切 CPU。

```bash
sudo nano /etc/safeguard/server.env
```

设置：

```env
SAFEGUARD_OCR_USE_GPU=false
```

然后：

```bash
conda activate scanocr
python -m pip uninstall -y paddlepaddle-gpu paddlepaddle
python -m pip install paddlepaddle
sudo systemctl restart safeguard-ocr
```

### 14.6 Admin Token 是中文导致前端 fetch 报错

处理：

```bash
openssl rand -hex 32
sudo nano /etc/safeguard/server.env
sudo systemctl restart safeguard-api
```

确保 token 只包含 ASCII。

### 14.7 gRPC 大文件上传失败

现象：

```text
RESOURCE_EXHAUSTED
Received message larger than max (4194413 vs. 4194304)
```

原因：旧版本 Agent 4 MiB 分块加 protobuf 包装超过 gRPC 默认 4 MiB。

处理：

```bash
cd /opt/safeguard
git pull
sudo systemctl restart safeguard-api
```

Agent 侧重新运行最新代码或重新打包 EXE。当前代码：

- Server gRPC message limit：64 MiB。
- Agent gRPC message limit：64 MiB。
- Agent 默认 chunk：3 MiB。

### 14.8 Agent 配置仍然是旧 IP

处理：

```powershell
Get-Content "C:\ProgramData\SafeGuardAgent\install_config.json"

[Environment]::SetEnvironmentVariable("SAFEGUARD_SERVER_BASE", $null, "Machine")
[Environment]::SetEnvironmentVariable("SAFEGUARD_GRPC_UPLOAD_TARGET", $null, "Machine")

.\SafeGuardAgent.exe remove
.\SafeGuardAgent.exe install --server-base "http://10.0.80.28" --work-dir "C:\ProgramData\SafeGuardAgent"
```

### 14.9 Server 看不到 Agent 注册

检查 Windows：

```powershell
Get-Service SafeGuardAgent
Get-Content "C:\ProgramData\SafeGuardAgent\logs\agent.log" -Tail 200
curl http://10.0.80.28/api/v1/health
```

检查 Linux：

```bash
tail -f /opt/safeguard/server/logs/server_v2.log
```

如果 Server 日志完全没有 `POST /api/v1/agents/register`，说明 Agent 没有打到这台 Server，优先查 IP、环境变量、防火墙。

### 14.10 敏感文件数量和敏感档案数量不一致

历史原因：检测按 `file_hash` 去重，而归档应按 `agent_id + file_path` 建档。旧逻辑可能把相同 hash 的不同路径合并为一个档案。

处理：

```bash
cd /opt/safeguard
git pull
sudo systemctl restart safeguard-api
```

然后访问敏感档案页面触发 backfill，或重新跑测试数据。

## 15. 当前限制与后续改进建议

当前限制：

- 轻量模式不适合大规模生产并发。
- API 与 gRPC 在同一进程，API 只能单 worker。
- OCR GPU 对硬件和 Paddle wheel 版本敏感。
- 前端和接口鉴权仍较简单。
- Agent 安装链路需要人工确认 UAC、环境变量和配置文件。
- PDF 原生高亮对扫描型 PDF 有限制。
- rename / delete / save-as 的语义仍有边界。

优先改进：

1. 拆分 gRPC upload server 为独立 systemd 服务。
2. 引入 PostgreSQL、Redis、MinIO 的标准生产部署路径。
3. 增加部署诊断脚本，一键检查端口、权限、模型、Python 路径、nginx。
4. 增加 Agent 安装器 UI 或统一配置文件校验。
5. 增加结构化日志和文件 hash 生命周期追踪。
6. 增加自动化测试覆盖上传、dedup、归档、OCR、rename/delete。
